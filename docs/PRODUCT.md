# The Vibe 产品文档

版本：v0.2  
更新时间：2026-05-22  
状态：基于当前代码的业务逻辑基线

## 1. 文档用途

本文档是 The Vibe 后续产品和代码迭代的参照文档。任何会改变用户体验、游戏规则、卡牌结构、阶段节奏、部署方式或安全边界的改动，都应同步更新本文档。

维护原则：

- 代码实现应尽量与本文档描述保持一致。
- 如果代码和文档冲突，先确认真实产品意图，再同时修正文档和代码。
- 新增功能前，先在本文档补充产品依据、规则影响和验收标准。

## 2. 产品定位

The Vibe 是一个面向情侣或亲密伴侣的双人私密互动卡牌网页。产品核心不是传统桌游胜负，而是通过“设定边界、共同确认、轮流抽牌、温度升高”的流程，帮助两个人在安全共识下逐步进入更亲密的交流和互动状态。

当前产品形态：

- 静态网页应用，基于 React + Vite 构建。
- 移动端优先，PC 端可访问。
- 无账号系统，无服务端业务逻辑，无数据库。
- 当前游戏状态仅存在于浏览器内存中，刷新页面会丢失本局进度。

核心价值：

- 用仪式感降低亲密沟通的尴尬。
- 用温度阶段控制互动强度。
- 用“随时跳过”的机制保护双方边界。
- 用视觉、动效和触感反馈增强氛围。

## 3. 目标用户和场景

目标用户：

- 已建立亲密关系的成年伴侣。
- 希望通过轻量卡牌互动增强情绪连接、表达欲和亲密感的用户。

主要使用场景：

- 私密空间中的双人互动。
- 手机浏览器打开后直接体验。
- 一局从轻度话题开始，逐步进入更高温度阶段。

非目标场景：

- 多人聚会游戏。
- 公开场合展示。
- 需要账号、社交分享、排行榜或云端存档的游戏。

## 4. 产品原则

1. 同意优先  
   游戏必须建立在双方自愿基础上。当前产品通过“契之誓”双人确认来表达进入游戏前的共识。

2. 随时跳过  
   任意卡牌都应允许用户拒绝执行。当前实现中，向左滑会进入惩罚缓冲区；用户可以完成替代任务，也可以直接跳过并承受轻微降温。

3. 渐进升温  
   游戏不是随机进入高强度内容，而是通过温度值逐步解锁更高阶段卡牌。

4. 氛围主导  
   视觉、文案和动画服务于沉浸感，但不应影响核心操作流畅性。动画性能优先级高于复杂装饰。

5. 本地隐私  
   当前版本不上传玩家姓名、卡牌选择或互动记录。

## 5. 当前页面流程

当前应用由五个主屏幕组成：

```text
landing -> setup -> oath -> board -> afterglow -> landing
```

### 5.1 Landing 首页

代码入口：`src/components/LandingView.tsx`

用户看到品牌、产品氛围、进入按钮和帮助弹窗。

主要功能：

- 展示产品名 The Vibe。
- 点击“进入”进入基础设定页。
- 点击右上角帮助按钮查看玩法简介。
- 页面底部展示 Consent Protocol 安全提示。

### 5.2 Setup 基本设定

代码入口：`src/components/SetupView.tsx`

用户在进入游戏前配置双方身份和尺度。

配置项：

- 玩家 A 昵称，默认“她 / A”。
- 玩家 B 昵称，默认“他 / B”。
- 玩家 A/B 的角色：主导者或配合者。
- 深度刻度：微温、灼热、沸腾。

角色规则：

- A 和 B 必须互为相反角色。
- 选择 A 为主导者时，B 自动成为配合者。
- 选择 B 为主导者时，A 自动成为配合者。

深度刻度规则：

- `level1`：只使用 Phase 1 卡牌。
- `level2`：使用 Phase 1 + Phase 2 卡牌。
- `all`：允许使用 Phase 1 + Phase 2 + Phase 3 卡牌。

注意：

- 深度刻度是卡池硬限制。`level1` 卡池中不应出现 Phase 2 或 Phase 3；`level2` 卡池中不应出现 Phase 3。
- 温度阶段是抽牌优先级。即使选择 `all`，Phase 3 也必须等到温度达到 80 且双方完成共燃确认后再出现。

### 5.3 Oath 契之誓

代码入口：`src/components/OathView.tsx`

这是进入游戏前的双方确认仪式。

核心规则：

- 两位玩家需要同时按下各自的指纹按钮。
- 只有双方同时按住时，进度条才会增长。
- 任意一方松开，且尚未完成时，进度归零。
- 进度达到 100 后，播放完成动效，并进入游戏桌面。

当前誓约准则：

- 绝对自愿。
- 随时叫停。
- 轻柔交互。

### 5.4 Board 游戏桌面

代码入口：`src/components/BoardView.tsx`

这是游戏主体验页面，包含抽牌、翻牌、滑动完成/跳过、温度阶段、惩罚缓冲区、共燃确认和阶段升级动画。

主要状态：

- `deck`：当前剩余卡池。
- `activeCard`：当前抽到的卡。
- `currentTurn`：当前回合玩家，A/B 交替。
- `temperature`：当前局内温度，范围 0-100。
- `isFlipped`：当前卡牌是否已翻开。
- `showPunishModal`：是否显示惩罚缓冲区。
- `activeStageUpgrade`：是否显示阶段升级过场。

主流程：

```text
等待抽牌
-> 当前玩家点击牌堆
-> 从 deck 中抽取符合阶段规则的卡牌
-> 卡牌自动翻开
-> 玩家向右滑：完成卡牌
-> 玩家向左滑：进入惩罚缓冲区
-> 处理完成后切换回合
-> deck 为空或本轮已处理 80 张卡牌时进入 afterglow
```

### 5.5 Afterglow 结束页

代码入口：`src/components/AfterglowView.tsx`

当卡池抽完，或本轮已处理 80 张卡牌后进入结束页。

主要功能：

- 展示结束文案。
- 播放慢速氛围动效。
- 延迟展示温柔返回按钮。
- 点击返回后回到 Landing，并重置游戏。

## 6. 温度和阶段规则

温度是当前产品的核心进度系统，同时驱动游戏规则和背景氛围。

阶段划分：

| 温度范围 | 阶段 | 标签 | 含义 |
| --- | --- | --- | --- |
| 0-39 | Phase 1 | 微温 | 破冰、走心、低强度互动 |
| 40-79 | Phase 2 | 灼热 | 更强的感官和亲密互动 |
| 80-100 | Phase 3 | 沸腾 | 最高强度互动 |

当前实现细节：

- `BackgroundGlow` 和 `BoardView` 共享 `40 / 80` 阶段阈值，避免背景氛围和游戏阶段错位。
- `BackgroundGlow` 根据温度切换背景色、光效速度和氛围强度。
- `BoardView` 根据温度切换温度条、阶段标签和阶段解锁。
- 完成卡牌会增加温度，最大不超过 100。
- 完成惩罚不会增加温度，也不会降低温度。
- 直接跳过会降低 5 点温度，最低不低于 0。
- 从 Phase 1 进入 Phase 2 时，会触发短促升温动效。
- 从 Phase 2 进入 Phase 3 前，必须完成双方共燃确认；确认后才触发 Phase 3 全屏过场。
- 卡牌库中的 `级别4` 不新增第四个视觉阶段；它属于 Phase 3 的隐藏延展卡池，只在完整模式、Phase 3 已完成双方确认、且温度达到 100 后才允许抽出。

当前版本已统一游戏初始温度为 0。

## 7. 卡牌系统

代码入口：`src/data/cards.ts`

当前卡牌数据直接写在 TypeScript 文件中。

### 7.1 Card 数据结构

每张卡牌包含：

```ts
interface Card {
  id: string;
  phase: 1 | 2 | 3;
  level?: 1 | 2 | 3 | 4;
  type: 'topic' | 'game' | 'action' | 'play';
  title: string;
  content: string;
  initiatorPrompt: string;
  receiverPrompt: string;
  points: number;
}
```

字段含义：

- `id`：唯一标识，不应重复。
- `phase`：所属温度阶段。
- `level`：卡牌库强度级别；未填写时默认等于 `phase`。从飞书导入 `级别4` 时，应写为 `phase: 3, level: 4`。
- `type`：卡牌类型，当前包括走心、默契、触碰、玩乐。
- `title`：卡牌标题。
- `content`：双方共同看到的核心任务。
- `initiatorPrompt`：主导者看到的角色提示。
- `receiverPrompt`：配合者看到的角色提示。
- `points`：完成后增加的温度值。

### 7.2 当前卡牌数量

当前源码中的实际业务卡牌：

- 主卡牌：108 张。
- Phase 1：36 张。
- Phase 2：10 张。
- Phase 3：62 张，其中 `level: 4` 延展卡 52 张。
- 惩罚项：48 条。

### 7.3 抽牌规则

开局时根据用户选择的深度刻度过滤卡池：

- `level1`：仅 Phase 1。
- `level2`：Phase 1 + Phase 2。
- `all`：Phase 1 + Phase 2 + Phase 3，并保留 `level: 4` 的隐藏延展卡。

过滤后会进行一次 Fisher-Yates 随机洗牌，避免开局卡组顺序固定。

每次抽牌时再根据当前温度决定允许抽出的阶段：

- 温度小于 40：只优先抽 Phase 1。
- 温度达到 40：允许抽 Phase 1 + Phase 2。
- 温度达到 80 且深度刻度为 `all`，并完成共燃确认：进入 Phase 3 沸腾状态，只允许抽 `level: 3` 卡牌，不再抽 `level: 1` 或 `level: 2`。
- 温度达到 100 且深度刻度为 `all`，并完成共燃确认：允许抽 `level: 3` + `level: 4` 卡牌；未达到 100 前不能抽出 `level: 4`。

Phase 3 激活后，低级别卡牌会从候选集合中退出，避免高强度氛围被低强度问答打断。若温度因为直接跳过回落到 80 以下，则按当前温度重新回到对应候选集合。

每一次抽牌不是固定取 deck 中第一张牌，而是先计算当前已解锁、未抽过的候选集合，再从候选集合中随机抽取一张。这样在同一大卡池下，新局和旧局不容易出现完全相同的体验顺序。

单局最多处理 80 张卡牌。完成卡牌、完成惩罚或直接跳过都会计入本轮已处理数量；达到 80 张后直接进入结束流程，即使 deck 中仍有剩余卡牌。

如果当前 deck 中没有符合阶段的卡，不应 fallback 抽取更高阶段卡牌；若此时达到 Phase 3 条件，则先进入共燃确认，否则进入结束流程。

### 7.4 卡牌完成和跳过

完成：

- 向右滑动卡牌。
- 温度增加 `points`。
- 当前卡牌关闭。
- 回合切换。

左滑：

- 向左滑动卡牌。
- 进入惩罚缓冲区。
- 可以随机惩罚或换一个惩罚。
- 确认完成惩罚后关闭当前卡牌，不加温度、不扣温度。
- 直接跳过会降低 5 点温度。
- 回合切换。

当前产品语义提醒：

- 帮助文案强调“双方可以无条件自主拒绝执行”。
- 完成惩罚是替代路径，不应比完成原卡更有收益。
- 直接跳过保留安全出口，但会轻微降低温度。

## 8. 惩罚缓冲区

代码入口：`src/data/cards.ts` 和 `src/components/BoardView.tsx`

惩罚项结构：

```ts
interface PenaltyTask {
  id: string;
  text: string;
}
```

当前规则：

- 左滑卡牌后显示惩罚缓冲区。
- 惩罚题库只放惩罚任务，不混入真心话或主卡互动任务。
- 点击随机惩罚后，系统短暂随机切换并确定一个惩罚项。
- 用户可以继续点击“换一个”重新随机。
- 点击“确认完成”后进入下一回合，不加温度、不扣温度。
- 点击“直接跳过 -5”后进入下一回合，温度降低 5。

产品边界：

- 惩罚应保持轻量、可接受、可拒绝。
- 惩罚不应变成对用户跳过权的惩罚性压力。
- 如果未来新增惩罚内容，应遵循与卡牌同等的安全边界。

## 9. 玩家和回合规则

玩家模型：

```ts
interface Player {
  name: string;
  role: 'initiator' | 'receiver';
}
```

当前规则：

- 游戏只有 A/B 两名玩家。
- 默认 A 先手。
- 每张卡牌处理完成后切换回合。
- 当前回合玩家显示为“呼叫”，另一名玩家显示为“响应”。
- 角色提示根据玩家的 `role` 决定，不直接等同于当前回合。

说明：

- “当前回合”决定谁抽牌、谁被呼叫。
- “主导者/配合者”决定看到哪一组角色提示。
- 如果未来要让主导者随回合切换，需要改动当前模型。

## 10. 视觉和交互系统

### 10.1 背景氛围

代码入口：`src/components/BackgroundGlow.tsx`

背景根据温度分三档变化：

- Phase 1：柔和、慢速、粉橘色系。
- Phase 2：更高亮、更快、更偏红色。
- Phase 3：更强烈、更快、更偏紫红色，并出现心跳式边缘光。

用户 30 秒无操作后，`useIdleTime` 会让背景动画变慢。

### 10.2 卡牌交互

核心动作：

- 点击牌堆抽牌。
- 卡牌延迟自动翻开。
- 向右滑完成。
- 向左滑跳过。

辅助反馈：

- 翻牌粒子。
- 卡牌光晕。
- 滑动方向水印。
- 支持浏览器震动 API 的设备会触发轻微触感反馈。

### 10.3 性能要求

当前体验高度依赖动画和模糊光效。后续优化时应优先保证：

- 卡牌拖动时保持流畅。
- 阶段升级过场不阻塞主线程。
- PC 和移动端均不出现明显卡顿。
- 复杂视觉层应在低性能设备上可降级。
- 翻牌只保留一个主动作和少量情绪反馈，不叠加多层常驻光效。
- 拖拽时暂停卡牌内部追光、粒子和呼吸光。
- Phase 2 升级使用短促升温反馈；Phase 3 打开使用心跳裂隙承接，避免巨大旋转 blur 和高频闪烁。

## 11. 内容扩展规范

如果将卡牌扩展到约 200 张，静态网页本身可以承载，但需要调整内容组织和单局长度控制。

建议规则：

- 不再把全部卡牌堆在单个 `cards.ts` 文件中。
- 按阶段拆分，例如 `phase1Cards.ts`、`phase2Cards.ts`、`phase3Cards.ts`。
- 飞书表格中的 `级别4` 应作为 Phase 3 的 100° 后延展卡池导入，不应创建新的温度计阶段。
- 每局不要求抽完整个 200 张卡池，系统硬性限制单局最多处理 80 张。
- 每次抽牌从当前已解锁候选集合中随机抽取，避免大卡池仍呈现固定顺序。

新增卡牌验收标准：

- `id` 唯一，且能看出阶段和序号。
- `phase` 与内容强度一致。
- `level` 与飞书表格的卡牌级别一致；`level: 4` 必须搭配 `phase: 3`，并遵守 100° 解锁规则。
- `type` 使用现有枚举，除非产品文档先扩展新类型。
- `content` 是双方共同任务，不偏向某一方。
- `initiatorPrompt` 和 `receiverPrompt` 必须同时存在。
- `points` 应匹配阶段节奏，避免单张卡过快跨阶段。
- 文案必须保留自愿、可拒绝、轻柔互动的边界。

当前 points 参考：

- 走心 `topic`：默认 +6。
- 默契 `game`：默认 +8。
- 触碰 `action`：默认 +9。
- 玩乐 `play`：默认 +10。

## 12. 当前技术边界

构建和部署：

- 使用 Vite 构建。
- 构建产物位于 `dist/`。
- 当前依赖要求较新的 Node.js，部署时建议使用 Node.js 22。
- 可部署到 CloudBase 静态托管。

数据和状态：

- 无后端。
- 无持久化。
- 无用户身份。
- 无在线同步。
- 无埋点。

当前未充分使用的结构：

- `types.ts` 中定义了 `GameState`，但当前业务状态主要分散在多个组件的 `useState` 中。
- `history` 字段未被当前流程使用。
- `discardedCount` 会递增，并用于控制单局 80 张上限；当前界面没有展示进度或历史。

## 13. 后续产品债务

这些不是必须立刻修复的问题，但会影响后续扩展。

1. 统一初始温度  
   `App` 和 `BoardView` 的初始温度存在 20 与 15 的差异。

2. 单局长度可配置  
   当前逻辑已限制单局最多 80 张；后续如需按模式设置不同上限，可把该值从常量升级为配置。

3. 数据结构拆分  
   卡牌增多后应拆分文件或改为可导入的数据源。

4. 跳过权一致性  
   当前“无条件拒绝”和“跳过后必须轮盘”之间存在产品语义张力。

5. 游戏进度展示  
   当前没有展示已完成数量、剩余数量、历史记录或本局摘要。

6. 动画降级策略  
   需要为 PC 端卡顿和低性能设备定义降级方案。

7. 状态集中化  
   如果后续规则继续变复杂，可以考虑将游戏状态集中为 reducer 或状态机。

## 14. 视觉与动效设计原则

本轮 v2 视觉升级的目标是让规则变化不仅可理解，也能被感受到。整体继续沿用暗色玻璃、玫瑰红、琥珀、紫红流光的品牌气质，但动效表达从单纯炫光推进为更有情绪节奏的体验：微温是呼吸，灼热是靠近，沸腾是共振打开。

设计关键词：

- 私密、柔软、克制、流动。
- 情绪递进，而不是功能跳转。
- 双人共同确认，而不是单人点击授权。
- 动效服务感受，不做无意义装饰。
- 强氛围但不卡顿，尤其卡牌拖拽必须轻盈。

### 14.1 模式化温度计

温度计随深度模式改变展示方式：

- `level1` 微温模式：只显示连续升温流，不显示灼热或沸腾节点；温度到 100 代表本局微温氛围完成，不代表解锁更高尺度。
- `level2` 灼热模式：显示微温、灼热两段路径；进入灼热时触发短促升温动效。
- `all` 完整模式：显示微温、灼热、沸腾三段路径；沸腾节点在双方确认前保持锁定微光状态。

### 14.2 共燃确认

完整模式进入 Phase 3 前，必须出现双人二次确认仪式。它和开局的“契之誓”形成呼应，但不重复使用进度条。

交互结构：

```text
玩家 A 指纹      中央心跳共振线      玩家 B 指纹
```

规则：

- 双方同时长按时，左右两侧心跳波形向中心同步，并推进解锁。
- 任意一方松开时，心跳波形快速降频回落。
- 中央心形跟随共振节奏跳动，完成时心跳线变成打开裂隙。
- 完成后显示“双方已确认，沸腾阶段开启”，并衔接 Phase 3 升级动画。
- 用户也可以选择“停留在灼热阶段”，本局不再进入 Phase 3。

### 14.3 四类卡牌视觉语言

卡牌类型扩展为四类：

- `topic` 走心：柔粉、淡琥珀，细线和慢淡入。
- `game` 默契：玫瑰粉与暖白，双点同步和轻微回弹。
- `action` 触碰：玫瑰红、暖橙，边缘光更贴近卡面。
- `play` 玩乐：紫红、莓果色、少量金色高光，更灵动但不派对化。

阶段决定尺度，类型决定玩法。玩乐不强制只属于 Phase 3，但 Phase 3 可以包含更高强度的玩乐卡。

### 14.4 惩罚缓冲区

左滑后的界面不表达失败，而表达“柔软的转圜空间”。

固定操作：

- `随机惩罚 / 换一个`：随机展示一条替代任务。
- `确认完成`：完成当前惩罚后切换回合，不加温度、不扣温度、不计入阶段完成数。
- `直接跳过 -5`：不完成原卡也不完成惩罚，温度降低 5，最低不低于 0。

视觉上应压暗背景但避免警报式红色；直接跳过按钮需要明确但不羞辱用户。

### 14.5 动效性能原则

- 所有重要动效都必须有情绪原因：确认、靠近、升温、打开、回落。
- 拖拽、翻牌、按钮点击优先流畅，不能被背景动效影响。
- 避免持续高频闪烁。Phase 3 可以强，但要像心跳，不像警报。
- 阶段过场只允许一个主视觉动作，背景光场作为辅助，不能叠加多层旋转和大面积动态 blur。
- 支持减少动态效果时，应降级为淡入、亮度变化和短位移动效。
